<!-- SLIDE: 1 — Cover -->

The Task tool is the mechanism behind everything we covered in 3.4, 3.5, and 3.6. When we say "the coordinator spawns a subagent," what we mean concretely is "the coordinator calls the Task tool." This lecture covers what the Task tool is, its key parameters, the allowedTools requirement the exam tests, and the least-privilege principle. Get this right and the multi-agent patterns all fall into place.

<!-- SLIDE: 2 — What the Task Tool Is -->

Let's anchor what the Task tool actually is.

It does four things. It creates an independent Claude instance. It passes a task description and instructions. It constrains the subagent to specific tools. And it returns the subagent's final output to the coordinator.

What it is NOT: it is NOT a generic HTTP call to another service. It is NOT a thread or process fork. It is NOT a shared-memory communication channel. And it is NOT persistent — each call is independent.

The analogy I like: the Task tool is Claude's native way of saying "I need a specialist to handle this — let me formally assign the work." That framing keeps you from reaching for web requests or shared state when what you actually want is a specialized, isolated Claude instance.

<!-- SLIDE: 3 — The Task Tool — Key Parameters -->

The Task tool takes three key parameters.

Description: a short label for what this subagent is doing. Think of it as a log line, not a specification.

Prompt: the complete, self-contained task specification — objective, context, output format, constraints. This is all the subagent knows. Because of context isolation from 3.6, there is no backchannel. If it's not in the prompt, the subagent doesn't have it.

And allowedTools: the list of tools the subagent is permitted to use.

Two things the exam tests here. One: the prompt must be self-contained, because it's everything the subagent knows. Two: allowedTools must include "Task" or the subagent can't accept the assignment. That second point is the next slide.

<!-- SLIDE: 4 — The allowedTools Requirement -->

This is one of the highest-value slides in Domain 1.

The allowedTools parameter MUST include "Task". If it's missing, the subagent cannot properly accept its task specification. The reason is mechanical: the Task tool is how an agent formally accepts an assignment. The subagent needs Task in its allowed set in order to process its task specification. Without it, it can't acknowledge and execute.

The exam will hand you an answer where allowedTools lists everything except "Task," and ask why the subagent isn't doing what's expected. The answer is right there in the parameter list. This is a classic almost-right distractor — the list looks reasonable, read_file and search_documents and whatever else the subagent needs are all present, but "Task" is missing and nothing works. Recognize the pattern on sight.

<!-- SLIDE: 5 — Constraining Subagent Capabilities via allowedTools -->

Beyond the required "Task," the rest of allowedTools is where you apply the principle of least privilege.

Each subagent only gets the tools required for its specific job. Document analysis doesn't need web search. Web search doesn't need file write. The research subagent doesn't need deployment tools.

Why it matters: an overpowered subagent can take unintended actions. Give write access where read-only was meant and you get data corruption. Give deployment access to an investigative subagent and you get a production incident. Constrained allowedTools means predictable, auditable, safer behavior.

Three quick examples. Research: Task, web_search, read_url. Report writer: Task, write_file. Code reviewer: Task, read_file, run_tests. Narrow lists. No "just in case" tools.

<!-- SLIDE: 6 — Coordinator Spawning Multiple Subagents in One Turn -->

Here's what parallel spawning looks like at the Task-call level.

The coordinator emits a single response whose content array contains two tool_use blocks — both with name "Task." One spawns the research subagent. The other spawns the document subagent. Each has its own description, its own prompt, and its own allowedTools list — each following least privilege.

Both Task calls are in the same coordinator response turn. That's what "parallel" means here. The coordinator then waits for both Task results before continuing. We'll dive into the full parallel execution pattern in the next lecture, 3.8 — but recognize the structural shape now: multiple Task blocks in one response equals parallel.

<!-- SLIDE: 7 — Exam Tip: Two Task-Tool Traps -->

Two exam traps to watch for.

Trap one: the subagent isn't processing its assignment correctly — and the root cause is that "Task" is missing from allowedTools. The question describes unexpected behavior, and the fix is a single missing entry in a list.

Trap two: an answer that hands every subagent access to all tools "for flexibility." That violates least privilege. The correct answer narrows the list. Every unnecessary tool expands the blast radius of a misbehaving subagent.

The rule the exam is looking for: every subagent's allowedTools must include "Task" plus only the tools needed for its specific job. Nothing more. Domain 1 is twenty-seven percent of the exam and this pattern shows up in Scenario 3 — Multi-Agent Research — which is one of the six scenarios you can't skip.

<!-- SLIDE: 8 — Takeaways -->

Six things to carry into 3.8. The Task tool is the mechanism coordinators use to spawn independent subagent instances. allowedTools must include "Task" — without it, the subagent cannot properly accept its assignment. Each Task call creates a fresh independent instance — no state persists. The prompt must be self-contained — it's all the subagent knows. Apply least privilege — each subagent gets only the tools needed for its specific job. And multiple Task calls in one coordinator response equals parallel spawning.

Next up, 3.8, we take that last point and go deep. What parallel subagent execution actually is structurally, when to choose it over sequential, and the single trap the exam plants around the word "parallel."
