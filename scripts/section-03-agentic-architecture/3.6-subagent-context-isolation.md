<!-- SLIDE: 1 — Cover -->

In 3.4 and 3.5 we saw that subagents don't inherit the coordinator's history. This lecture goes deep on that rule. What isolation actually means. What the subagent does and doesn't have. And the design implications of both. Context isolation is the mechanism that makes multi-agent systems work. If you don't understand it, you'll pick distractors that assume implicit propagation — and that's one of the most frequently tested traps in Domain 1.

<!-- SLIDE: 2 — What Context Isolation Actually Means -->

Let's be precise.

The subagent does NOT have the coordinator's conversation history. It does NOT have results from other subagents. It does NOT have the original user request unless the coordinator passes it. And it does NOT have any context from previous turns.

The subagent DOES have its own system prompt if one is provided. It has the task prompt the coordinator constructed. It has the tools in its allowedTools list. And it has only what the coordinator explicitly passed — nothing more.

The design principle here is predictability plus security. A subagent that can't accidentally SEE unrelated data can't accidentally ACT on it. That's the whole point. Isolation isn't an inconvenience — it's a feature.

<!-- SLIDE: 3 — Context Isolation in Code — What the Subagent Receives -->

Here's what this actually looks like in code.

The coordinator has rich context. The original user objective. Facts from the research subagent. Notes from the document subagent. None of that flows automatically to the next subagent.

The coordinator has to serialize everything the synthesis subagent needs into the task prompt. Objective. Facts. Notes. Output format. Then the Task call goes out with that self-contained prompt plus a narrow allowedTools list.

The subagent sees only that prompt. Every fact it needs must be injected by the coordinator. That's the contract. If you forget a piece, the subagent either asks for information it doesn't have, produces output that doesn't fit, or calls a tool to re-collect data that was already gathered upstream.

<!-- SLIDE: 4 — The Explicit Passing Requirement -->

Three rules about what to pass.

What to pass: the original user objective, background facts or data, output format, constraints, and results from prior subagents that this specific subagent needs. Everything relevant. Explicitly.

What fails without it: the subagent produces output that doesn't fit the broader task, or it re-collects information via tools — wasting resources, wasting time, and sometimes producing inconsistent results because the re-collection differs from what the coordinator already has.

What to NOT pass: don't dump the entire coordinator history. Pass only what's relevant. Irrelevant context increases cost and confuses the subagent. Big prompts are not better prompts.

<!-- SLIDE: 5 — Isolation — Benefits and Challenges -->

Isolation has clear benefits and real costs.

Benefits: predictability — behavior depends only on what you gave the subagent. Security — sensitive data doesn't leak between subagents. Parallelism — isolated subagents safely run concurrently, which is what makes the patterns in 3.8 possible. Testability — you can test each subagent with known inputs and get reproducible output.

Challenges: context engineering burden — the coordinator must construct complete, accurate prompts. Serialization overhead — large data structures have to be formatted for prompts. And no shared state — all results flow through the coordinator.

The design implication is that isolation shifts complexity to the coordinator's prompt construction logic. That's the tradeoff. You pay a little in the coordinator to get safe, parallel, testable subagents.

<!-- SLIDE: 6 — Each Task Call Creates an Independent Instance -->

Three implications of "no session persistence."

Implication one: call the Task tool twice with the same subagent description and you get two independent instances. The second has no memory of the first. They don't share state.

Implication two: retry logic must reconstruct the full task context. You cannot resume a failed subagent — you spawn a fresh one with the same, or corrected, prompt. That's not a bug. It's how isolation works.

Implication three: the coordinator is the only persistent entity. It accumulates results across subagent calls and maintains workflow state.

Mental model: subagents are functions. Call them with arguments. They return results. They don't remember between calls.

<!-- SLIDE: 7 — Exam Tip: Context Isolation Traps -->

Two traps to recognize on the exam.

Trap one: an answer that assumes the subagent "already knows" the user's original request because the coordinator received it. No. The subagent only knows what's in its task prompt. If the coordinator didn't inject the user's objective, the subagent doesn't have it.

Trap two: an answer that describes the subagent being "updated" or "continuing" from a prior call. No. There's no memory between Task calls. Two calls equals two instances.

The rule is short. Subagents start with a blank slate. The coordinator is the sole source of context for each subagent invocation. Everything must be passed explicitly. That one rule answers almost every isolation question — and the distractors here are classic almost-right: they read plausibly if you assume an ambient shared memory that doesn't exist.

<!-- SLIDE: 8 — Takeaways -->

Six things to carry forward. Subagents do NOT inherit the coordinator's history — each starts with a blank context. Each Task call creates an independent instance — no session persistence. Explicit context passing is mandatory — everything in the task prompt. Pass RELEVANT context only — dumping full history is inefficient and counterproductive. The coordinator is the only persistent entity — it accumulates results and maintains state. And isolation enables parallelism, predictability, and security at the cost of shifting complexity to the coordinator's prompt construction.
