<!-- SLIDE: 1 — Cover -->

In 3.4 we sketched the hub-and-spoke topology. Now we zoom into the hub itself — the coordinator. This lecture walks through its three jobs in detail: how to decompose a task well, how to delegate with explicit context, and how to aggregate results across every possible outcome. Get this right and your multi-agent system is debuggable and predictable. Get it wrong and it silently returns partial nonsense.

<!-- SLIDE: 2 — The Coordinator Is the Brain -->

The coordinator is the brain of a multi-agent system. It does five things. It receives the original user task. It decides how to split it into subtasks. It selects which subagents handle which parts. It passes the right context to each subagent. And it collects and synthesizes results.

What it does NOT do is the actual domain work. It doesn't browse the web. It doesn't analyze documents directly. It doesn't execute code. It doesn't write the final research report. Those are subagent responsibilities.

The exam implication is direct. If an answer choice has the coordinator executing domain tasks instead of delegating, it's wrong. The coordinator orchestrates — it never operates. This is Domain 1 territory, twenty-seven percent of the exam, and Scenario 3 — Multi-Agent Research — lives on this distinction.

<!-- SLIDE: 3 — Decompose — Breaking the Task Well -->

Decompose is the first job, and the easiest to get wrong. Poor decomposition bundles responsibilities — a research agent that does research AND writes the summary, subtasks with unclear boundaries, or one subagent implicitly depending on another's output format.

Good decomposition gives each subagent one clearly defined job. Independent tasks can run in parallel. Dependent tasks are chained explicitly. Each subtask has a clear output contract.

And the coordinator doesn't have a fixed map of subtask-to-subagent. Dynamic selection means the coordinator chooses the right specialized subagent based on the nature of the subtask — which requires it to reason about capabilities. Not every task needs the same agent. The research subagent handles research. The document analysis subagent handles documents. The coordinator picks.

<!-- SLIDE: 4 — Delegate — Explicit Context Passing -->

Delegate is the second job. Three rules.

Rule one: subagents start with fresh context. The coordinator's conversation history is NOT passed down. Everything the subagent needs must be in the task prompt.

Rule two: the task prompt must include everything — objective, relevant background, output format, constraints. If the synthesis subagent needs facts from the research pipeline, the coordinator injects them explicitly. Nothing flows automatically.

Rule three: least-privilege tools. Only give each subagent the tools it needs. Document analysis doesn't need web search. Constrained tools make for predictable, auditable behavior.

The failure mode here is assuming the subagent "knows what you mean." It doesn't. Make every assumption explicit.

<!-- SLIDE: 5 — Aggregate — Collecting and Synthesizing Results -->

Aggregate is the third job, and the one candidates underweight. Three outcomes to handle.

Happy path: all subagents return results. The coordinator synthesizes — combine, deduplicate, format into a single coherent output.

Partial failure: one subagent fails or returns no result. The coordinator decides — retry, proceed without, or surface the failure to the caller. This is a decision the coordinator owns, not something to paper over.

Total failure: a critical subagent fails. The coordinator surfaces that clearly — NOT a silently incomplete result. Silent partial output is one of the worst failure modes in a multi-agent system, because the caller doesn't know what's missing.

The aggregation rule is simple: the coordinator is the single point of error handling for all subagent outputs. Errors are caught here, not silently dropped.

<!-- SLIDE: 6 — The Full Coordinator Lifecycle -->

Put it all together and the coordinator has a clean lifecycle. Receive the user task. Decompose it into subtasks. Select subagents dynamically. Spawn them via the Task tool with explicit context. Receive subagent results. Aggregate. Return the final output.

At each step, a specific responsibility. Decompose gives each subtask one owner, clear scope, and an explicit output contract. Select means choosing the right specialized subagent for the subtask — not all tasks need the same agent. Spawn passes exactly what's needed, no more, no less. Aggregate handles errors, synthesizes outputs, and returns a coherent result.

That's the whole shape of what a coordinator is. Everything else in this section is detail on one of those four moves.

<!-- SLIDE: 7 — Exam Tip: Coordinator Responsibility Boundaries -->

Two traps the exam loves to plant in coordinator questions.

Trap one: the coordinator doing domain work — browsing the web, writing the report directly. That looks plausible because it reads like the coordinator is being thorough. Almost-right is the whole trap. Coordinators orchestrate; they do not operate.

Trap two: implicit context propagation — an answer that suggests the subagent "picks up where the coordinator left off" without explicit passing. That's not how isolation works. The subagent starts fresh every time.

Remember the rule. Coordinator equals decompose, delegate with explicit context, and aggregate with error handling. Dynamic subagent selection means the coordinator chooses the right agent based on the nature of the subtask. Six-pick-four means you can't skip Scenario 3 — so you have to know this cold.

<!-- SLIDE: 8 — Takeaways -->

Six things to carry forward. Coordinator orchestrates — it never executes domain tasks directly. Decompose into bounded subtasks with clear output contracts and no implicit dependencies. Delegate with explicit context passing — subagents start fresh, everything lives in the prompt. Aggregate handles every outcome: success, partial failure, total failure, never dropped silently. Dynamic subagent selection means choosing the right specialized agent for the subtask. And the coordinator is the single point of error handling for all subagent outputs.
