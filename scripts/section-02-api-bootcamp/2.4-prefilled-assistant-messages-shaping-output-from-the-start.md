## SLIDE 1: The Preamble Problem

**Visual**: Split screen. Left: a chat bubble showing Claude's response starting with "Sure! I'd be happy to help you extract that information. Here is the JSON you requested:" followed by actual JSON. Right: the same response starting cleanly with `{`. A red X on the left, a green checkmark on the right.

You've asked Claude to extract structured data and return JSON.

Instead you get: "Sure! Here's the JSON you asked for..."

Then the JSON.

That preamble is harmless in a chat app.

But in a production pipeline, it breaks your JSON parser immediately.

The fix is a technique called prefilling, and it's one of the most useful tools in your API toolkit.

Let's look at exactly how it works.

---

## SLIDE 2: What Is a Prefill?

**Visual**: A diagram of the messages array as a vertical list of labeled boxes. The boxes are: `user` (blue), `assistant` (green), `user` (blue), and then a final `assistant` box (green, highlighted with a glow) labeled "PREFILL — Claude continues from here." An arrow points from the bottom of the prefill box downward into a response stream.

The `messages` parameter in the API is just an array of turns.

Each turn has a `role` — either `"user"` or `"assistant"`.

Normally you end with a user turn, and Claude writes the next assistant turn.

A prefill means you end the array with an assistant turn instead.

[click]

You're essentially handing Claude a partial sentence and saying: finish this.

Claude will continue from that exact character, word for word.

It cannot add anything before the text you provided.

That's the power — you control the very first token of the response.

## SLIDE 3: When Prefilling Is the Right Tool

**Visual**: A four-item icon list on a clean white background. Each item has a small icon and a short label: (1) curly braces icon — "Force JSON output", (2) speech bubble with X — "Eliminate filler preamble", (3) code block icon — "Start a code fence", (4) theater mask icon — "Lock in a persona or voice".

Prefilling is useful in four common situations.

First: forcing structured output to start correctly, like JSON beginning with `{`.

Second: eliminating filler like "Certainly!" or "Great question!" from the response.

Third: forcing a code block to open immediately with the right language tag.

Fourth: locking in a character voice mid-conversation without breaking immersion.

[click]

All four of these share the same underlying mechanic.

You're not adding a special parameter.

You're using the structure of the `messages` array itself.

## SLIDE 4: The Code — JSON Extraction Prefill

**Visual**: Dark-background code editor panel showing the full Python example below, with syntax highlighting. Three lines are highlighted in amber: the prefill message dict, the `content` value `"{"`, and the line accessing `response.content[0].text`. A sidebar note reads: "Claude completes the JSON from here."

```python
import anthropic
import json

client = anthropic.Anthropic()

user_data = "Name: Alice Chen, Role: Staff Engineer, Team: Platform"

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=256,
    messages=[
        {
            "role": "user",
            "content": f"Extract the following into JSON with keys name, role, team:\n\n{user_data}"
        },
        {
            # Prefill: Claude will continue from this exact character
            "role": "assistant",
            "content": "{"
        }
    ]
)

# Re-attach the opening brace Claude was given, then parse
raw = "{" + response.content[0].text
result = json.loads(raw)
print(result)
```

Notice that the last message in the array has `role: "assistant"`.

That single opening brace is the prefill.

Claude continues from that character — it cannot add a preamble before it.

[click]

One important detail: the prefill text is NOT included in the response body.

Claude returns only what it generated after the prefill.

So when you reconstruct the full value, you prepend the prefill yourself — that `"{"` on the last line.

## SLIDE 5: Code Generation and Format Prefills

**Visual**: Two side-by-side code panels. Left panel shows a prefill of ` ```python\n ` with a label "Forces Python code block". Right panel shows a prefill of `"The answer is:"` with a label "Forces specific phrasing". Both panels have a downward arrow labeled "Claude continues here."

The same technique works for any output format you need to lock in.

For code generation, prefill with the opening fence and language tag.

```python
{
    "role": "assistant",
    "content": "```python\n"
}
```

Claude will now write Python code directly — no introduction, no explanation first.

[click]

For format control, prefill with the exact phrase you want to start the response.

```python
{
    "role": "assistant",
    "content": "The answer is:"
}
```

This eliminates any variation in how Claude opens its response.

It's especially useful when you're parsing outputs downstream with string operations.

## SLIDE 6: Prefills and Stop Sequences — The Power Pair

**Visual**: A flow diagram showing three stages left to right: a "Prefill" box (green, labeled "Controls the START"), a middle box showing Claude's generation in progress, and a "Stop Sequence" box (red, labeled "Controls the END"). An arrow runs through all three labeled "Clean, bounded output."

Prefills and stop sequences solve opposite halves of the same problem.

A prefill controls where Claude starts.

A stop sequence controls where Claude stops.

Together, they let you extract exactly what you want from a response.

[click]

Here's the pattern: prefill with an opening XML tag, stop at the closing tag.

```python
response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=512,
    stop_sequences=["</answer>"],
    messages=[
        {"role": "user", "content": "What is 12 * 8? Respond inside <answer> tags."},
        {"role": "assistant", "content": "<answer>"}  # prefill
    ]
)
# response.content[0].text contains only what's between the tags
```

The response will be the content between the tags — nothing before, nothing after.

That's a clean extraction with zero post-processing regex.

## SLIDE 7: Exam Tip

**Visual**: Amber callout box. Title: "Exam Tip". Icon: target/bullseye. Text: "There is no `prefill` parameter — it's just the messages array."

**Exam Trap**: Candidates look for a dedicated `prefill` parameter in the API or believe you need a special flag to enable this behavior.

**Correct Approach**: There is no such parameter. Prefilling works purely through the structure of the `messages` array. You add a message with `role: "assistant"` as the final entry, and Claude continues from its content. If the last message has `role: "user"`, Claude starts a fresh response. If it has `role: "assistant"`, Claude continues from exactly where you left off. That's the entire mechanic — know it cold.

## SLIDE 8: Key Takeaways

**Visual**: Clean bullet list on light background. Title: "What to Remember".

- A prefill is an `"assistant"` role message placed last in the `messages` array — Claude continues from its content, adding nothing before it
- The prefill text is not included in the response body — prepend it yourself when reconstructing the full output
- Common uses: force JSON to open with `{`, eliminate preamble, start code fences, lock in a persona
- Pair prefills with `stop_sequences` to bound output on both ends — the cleanest way to extract structured content
