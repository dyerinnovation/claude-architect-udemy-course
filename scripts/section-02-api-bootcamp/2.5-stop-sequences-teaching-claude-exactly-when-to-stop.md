## SLIDE 1: The Overgeneration Problem

**Visual**: A terminal window showing a model response. The relevant answer appears in the first three lines inside `<answer>` tags, cleanly formatted. Below that, separated by a faint line, is a large block of additional text — commentary, caveats, follow-up suggestions — all grayed out and stamped with "You didn't ask for this." A red scissors icon hovers at the line where the useful content ends.

You asked Claude a direct question and got exactly the right answer.

Then Claude kept going.

It added context, caveats, follow-up suggestions, and a closing summary.

In a chat interface, that feels helpful.

In a production system, it's noise your parser has to fight through.

Stop sequences let you draw a hard line in the output stream.

When Claude hits that line, it stops — immediately, cleanly, every time. This is another Scenario 6 tool — structured extraction pipelines lean on stop sequences to bound output cleanly.

## SLIDE 2: How Stop Sequences Work

**Visual**: A flow diagram showing Claude's token generation as a horizontal stream of words flowing right. Mid-stream, a red wall labeled with the stop sequence string `"</answer>"` blocks the stream. Above the wall: a green arrow labeled "Included in response." Beyond the wall: a gray section labeled "Never generated." Below the diagram, two callout boxes: `stop_reason: "stop_sequence"` and `stop_sequence: "</answer>"`.

The `stop_sequences` parameter takes an array of strings.

While Claude generates tokens, the API watches the output for exact matches.

The moment Claude produces one of your strings, generation halts.

The matched string itself is NOT included in the response body.

[click]

You can pass up to 8,191 stop sequences in a single request.

In practice you'll rarely use more than one or two.

When a stop sequence fires, `response.stop_reason` is `"stop_sequence"` — not `"end_turn"`.

And `response.stop_sequence` contains the exact string that matched.

Those two fields are your confirmation that the stop worked.

## SLIDE 3: When Stop Sequences Are the Right Tool

**Visual**: A four-item vertical list with icons. Each row has a small icon, a use case label in bold, and a one-line description beneath it. Items: (1) XML tag icon — "Structured extraction: stop at `</answer>` to get clean content", (2) delimiter icon — "Delimited output: stop at `###END###` with no trailing content", (3) dialogue bubble icon — "Multi-turn dialogue: stop at turn boundaries like `\nHuman:`", (4) code block icon — "Code generation: stop at a known end marker like `\n\`\`\`\n`".

Stop sequences shine in four common scenarios.

First: structured extraction — stop at a closing XML tag to get just the content inside.

Second: delimited output — use a custom marker to signal the end of relevant content.

Third: multi-turn structured dialogue — stop when you detect the next speaker's turn boundary.

Fourth: code generation — stop after the closing fence so you get one clean block.

[click]

The common thread is that you control the prompt, so you know what strings to expect.

You design the prompt to include a predictable stop marker.

Then you tell the API exactly what that marker is.

## SLIDE 4: The Code — Stop Sequences in Practice

**Visual**: Dark-background code editor panel showing the full Python example below with syntax highlighting. Four lines are highlighted: the `stop_sequences` parameter, the `stop_reason` check, the `stop_sequence` field access, and the final `print` of the extracted content. A sidebar annotation reads: "Check both fields — `stop_reason` tells you WHY it stopped, `stop_sequence` tells you WHAT matched."

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=512,
    # Halt generation the moment Claude outputs this exact string
    stop_sequences=["</answer>"],
    messages=[
        {
            "role": "user",
            "content": (
                "What is the capital of Japan? "
                "Wrap your answer in <answer> tags and nothing else."
            )
        }
    ]
)

# Confirm the stop sequence fired as expected
if response.stop_reason == "stop_sequence":
    matched = response.stop_sequence   # Contains "</answer>"
    # Content is everything inside the tags — the closing tag was never generated
    answer = response.content[0].text.strip()
    print(f"Extracted answer: {answer}")
    print(f"Stopped on: {matched}")
else:
    # Stop reason is "end_turn" — model finished before hitting the sequence
    print(f"Unexpected stop reason: {response.stop_reason}")
```

The response body contains everything Claude generated before the stop string.

The stop string itself was never appended — it's reported in `stop_sequence`, not the text.

[click]

The `else` branch matters in production.

If `stop_reason` is `"end_turn"`, Claude ran out of content before hitting your marker.

That usually means the prompt didn't produce the expected format.

Always handle both cases.

## SLIDE 5: Case Sensitivity and the Whitespace Traps

**Visual**: A three-column comparison table with a red/green header row. Column headers: "What You Wrote", "What Claude Generated", "Did It Stop?". Rows: (1) `"END"` / `"end"` / red X, (2) `"END"` / `"END"` / green checkmark, (3) `" END"` / `"END"` / red X (space before END in the sequence), (4) `"</Answer>"` / `"</answer>"` / red X. A bold caption below the table: "Stop sequences are exact string literals. Case and whitespace both count."

Stop sequences are exact string matches.

Not regex. Not fuzzy. Not case-insensitive.

Every character must match exactly — including capitalization and whitespace.

[click]

Here are the traps candidates hit most often.

`"END"` will not catch `"end"` — different case, no match.

`" END"` with a leading space will not catch `"END"` — different bytes, no match.

`"</Answer>"` will not catch `"</answer>"` — again, case mismatch.

The fix is simple: use stop sequences where you control the casing in the prompt.

XML tags are ideal because you write them in the prompt and you define the casing.

## SLIDE 6: Combining Stop Sequences with Prefills

**Visual**: A horizontal flow diagram. Left box (green): "Prefill: `<answer>`" with label "Controls START". Center box (gray): "Claude generates content here". Right box (red): "Stop sequence: `</answer>`" with label "Controls END". A bracket under all three labeled "Exactly what you wanted — nothing more." Below the diagram, a minimal code snippet showing both the prefill message and the `stop_sequences` parameter side by side.

Prefills and stop sequences are designed to work together.

The prefill forces Claude to open the format you expect.

The stop sequence forces Claude to close it exactly where you want.

```python
response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=256,
    stop_sequences=["</answer>"],   # stop at closing tag
    messages=[
        {"role": "user", "content": "What is 144 divided by 12?"},
        {"role": "assistant", "content": "<answer>"}   # prefill opening tag
    ]
)
# response.content[0].text is the raw answer, no tags, no commentary
```

The response is exactly the content between the tags.

No opening tag — that was the prefill, not in the response body.

No closing tag — that was the stop sequence, never generated.

Just the answer.

## SLIDE 7: Exam Tip

**Visual**: Amber callout box. Title: "Exam Tip". Icon: target/bullseye. Text: "Stop sequences are exact literals — `'END'` and `'end'` are different stop sequences."

**Exam Trap**: Candidates assume stop sequences behave like regex or case-insensitive pattern matching. They write a stop sequence in uppercase and wonder why it fires inconsistently when Claude generates a lowercase variant.

**Correct Approach**: Stop sequences are exact, case-sensitive string literals applied to the raw output stream before any post-processing. Whitespace is significant. To guarantee a match, use stop sequences that you also control in the prompt — XML closing tags are the safest choice because you define their exact casing when you write the prompt. Always verify the stop fired by checking `response.stop_reason == "stop_sequence"` and reading `response.stop_sequence`. A distractor that claims case-insensitive matching looks almost right — that's the trap.

## SLIDE 8: Key Takeaways

**Visual**: Clean bullet list on light background. Title: "What to Remember".

- `stop_sequences` is an array of up to 8,191 exact string literals — when any one matches Claude's output, generation halts immediately and the string is not included in the response body
- Check `response.stop_reason`: `"stop_sequence"` means it worked; `"end_turn"` means Claude finished without hitting your marker — always handle both
- Stop sequences are case-sensitive exact matches — whitespace counts, regex does not apply
- Pair with prefills for clean bounded extraction: prefill opens the format, stop sequence closes it, and the response contains exactly what you need
