## SLIDE 1: Where You Put Instructions Changes Everything

**Visual**: Two side-by-side chat windows. Left window: a customer support bot that forgets its persona mid-conversation. Right window: a bot that stays perfectly in character throughout. A banner above each reads "instructions in user message" vs "instructions in system prompt."

Here's a scenario you'll see on the exam.

A developer builds a customer support bot for an airline.

They put the bot's persona instructions in the first user message.

Three turns in, the bot starts answering like a general assistant — no persona, no constraints.

What went wrong?

The instructions were in the wrong place.

Understanding the `system` parameter is one of the most high-value things you can learn today. Scenario 1 — the Customer Support Resolution Agent from lecture 1.1 — lives and dies on this exact decision. And it feeds straight into Domain 4, which is twenty percent of your exam.

---

## SLIDE 2: What the System Parameter Actually Is

**Visual**: Diagram of a `messages.create()` call rendered as a layered structure. The `system` parameter floats above the `messages` array like a header. An annotation reads "applies to all turns." The messages array below shows multiple user/assistant alternating turns with no repeated instructions.

The `system` parameter is a top-level field in every `messages.create()` call.

It lives *outside* the `messages` array — above it, conceptually.

Think of it as a standing brief that Claude reads before the conversation begins.

It persists automatically across every turn in the conversation.

You write it once. It applies everywhere.

The `messages` array holds the conversation itself — the back-and-forth dialogue.

The `system` parameter holds the rules, persona, and constraints for that dialogue.

These are fundamentally different things and they belong in different places.

---

## SLIDE 3: The Right Way — System Parameter in Action

**Visual**: Dark-background Python code editor. The `system` parameter is highlighted with a yellow border. The `messages` array contains only user and assistant turns — no instructions repeated. A green checkmark badge in the top-right corner of the code block.

Here's what correct usage looks like.

```python
import anthropic

client = anthropic.Anthropic()

# System prompt defined once — applies to every turn
SYSTEM_PROMPT = """You are Aria, a friendly customer support agent for SkyLine Airlines.
You only answer questions about flights, baggage, and reservations.
If asked about anything else, politely redirect the customer."""

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=SYSTEM_PROMPT,       # Top-level parameter — NOT in messages array
    messages=[
        {
            "role": "user",
            "content": "Hi, I need to change my seat on flight SK442."
        }
    ]
)

print(response.content[0].text)
```

Notice the `system` parameter sits alongside `model` and `max_tokens` — not inside `messages`.

Aria's persona and constraints are established before the first user message.

Every subsequent turn in this conversation will operate under those rules.

You never repeat the system prompt in the messages array.

## SLIDE 4: The Wrong Way — Instructions in the User Message

**Visual**: Dark-background Python code with a red X badge in the top-right corner. The first message in the `messages` array has both instructions and a user question crammed together. An annotation arrow points to the `role: "user"` line with the label "this role is for user input, not instructions."

Now let's look at the pattern that breaks things.

```python
# WRONG: Embedding instructions inside the first user message
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    # No system parameter
    messages=[
        {
            "role": "user",
            # Instructions mixed into a user turn — this is the mistake
            "content": """You are Aria, a customer support agent for SkyLine Airlines.
Only answer questions about flights, baggage, and reservations.

Hi, I need to change my seat on flight SK442."""
        }
    ]
)
```

[click]

This works for exactly one turn.

When the conversation continues, you have a problem.

The instructions are now buried in the history as a user message.

They're not guaranteed to influence future turns the same way a real system prompt would.

And here's the critical difference: you're responsible for replaying the full `messages` array on every API call.

If you truncate the history to save tokens, you lose the instructions entirely.

A system prompt, by contrast, always comes through — it's never truncated from the history.

## SLIDE 5: Multi-Turn Behavior — System vs. User Message Instructions

**Visual**: Two-panel comparison. Left panel shows a multi-turn conversation where the system prompt is visually "pinned" above all turns like a header. Right panel shows a conversation where instructions were in turn 1, and turns 2 and 3 show the bot drifting. Annotations point to the drift in the right panel.

Let's see this play out across multiple turns.

```python
# How YOU maintain conversation history across turns
conversation_history = []

def chat(user_message):
    conversation_history.append({
        "role": "user",
        "content": user_message
    })

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="You are Aria, SkyLine Airlines support. Only discuss flights.",
        messages=conversation_history   # Full history sent every call
    )

    assistant_reply = response.content[0].text
    conversation_history.append({
        "role": "assistant",
        "content": assistant_reply
    })

    return assistant_reply

# Turn 1 — persona is active
chat("Can I change my seat?")

# Turn 2 — persona STILL active, because system prompt is always sent
chat("What about baggage fees?")

# Turn 3 — still Aria, still airline support only
chat("Can you help me book a hotel?")  # Aria will redirect — correctly
```

[click]

The system prompt is re-sent on every API call — you pass it every time.

But because it's a separate parameter, it's never confused with the conversation itself.

Even if you trim your `messages` history to save tokens, the system prompt stays intact.

This is exactly why the `system` parameter exists as a distinct field.

## SLIDE 6: When to Use System vs. User-Level Instructions

**Visual**: Two-column table on a white background. Column headers: "Use `system` parameter" and "Use user message content." Rows list scenarios: system column shows persona/role definitions, output format constraints, safety guardrails, tool usage policies. User message column shows one-off instructions, user-specific context for that turn, dynamic data injected per request.

Not every instruction belongs in the system prompt.

Use `system` for things that apply to the *entire conversation* from the start.

Persona and role definitions go here.

Output format constraints — like "always respond in JSON" — go here.

Safety guardrails and topic restrictions go here.

[click]

Use the `messages` array for things that are specific to a single turn.

If you need to inject dynamic data — like a customer's account details — that can go in the user message for that turn.

One-off instructions that only apply to the current question belong in `messages`, not `system`.

The rule of thumb: if it needs to be true for every single exchange, it belongs in `system`.

## SLIDE 7: Exam Tip

**Visual**: Amber callout box. Title: "Exam Tip". Icon: target/bullseye. Text: "System prompts persist. First user messages don't."

**Exam Trap**: Putting persistent role or persona instructions in the first user message instead of the `system` parameter, assuming they'll apply throughout the conversation.

**Correct Approach**: The `system` parameter is the only mechanism that guarantees instructions apply across all turns. User messages are conversation history — they can be truncated, and their influence on later turns is not guaranteed the same way.

Watch for exam scenarios where a bot "forgets" its persona or breaks its rules after several turns — the root cause is almost always instructions placed in a user message rather than the `system` parameter. Remember the six-pick-four: you don't know which scenarios show up, but Scenario 1 is one of the heaviest Domain-4 hitters, so skipping this one isn't an option.

## SLIDE 8: Key Takeaways

**Visual**: Clean bullet list on light background. Title: "What to Remember".

- The `system` parameter is a top-level field alongside `model` and `max_tokens` — it is never inside the `messages` array
- System prompts apply to every turn automatically — you pass the same `system` value on every API call, and it's never confused with conversation history
- Instructions placed in the first user message are part of the conversation history — they can be lost if history is truncated and don't behave like true system-level constraints
- Use `system` for persona, output format, safety rules, and anything that must be true for the entire conversation — use `messages` for turn-specific context and dynamic data
