## SLIDE 1: Why Does Your App Feel Slow?

**Visual**: Split screen. Left: a progress spinner frozen for 4 seconds, then text dumps all at once. Right: text appearing word by word, immediately. Bold caption: "Same model. Same response. Different delivery."

Have you ever watched a loading spinner for five seconds, then had a wall of text appear at once?

That's not a slow model — that's a slow *delivery* strategy.

Streaming is how you fix it.

Instead of waiting for Claude to finish the entire response, your app gets each word as it's generated.

Users see output immediately, and the experience feels completely different.

This lecture is about when to stream, when not to, and exactly what happens under the hood.

---

## SLIDE 2: What Is Streaming, Actually?

**Visual**: Animated diagram showing a single large box labeled "Full Response (blocking)" above a horizontal pipeline with many small boxes flowing out labeled `text_delta` chunks. Arrow from pipeline to a browser UI rendering text in real time.

Streaming uses a protocol called Server-Sent Events — SSE.

Instead of one big HTTP response, the server sends a series of small events over an open connection.

[click]

Each event carries a tiny piece of the response — a few tokens at a time.

Your code reads each event as it arrives and renders it immediately.

[click]

The total time for Claude to generate the full response is the same either way.

What changes is *when you see the first token* — that's called time-to-first-token, or TTFT.

Streaming dramatically reduces TTFT, which is why it *feels* faster even though it isn't.

## SLIDE 3: The Streaming Event Sequence

**Visual**: Vertical timeline with six labeled events in order, each on its own row with a color-coded badge. Rows: `message_start` (blue), `content_block_start` (green), `content_block_delta` (green, repeating), `content_block_stop` (green), `message_delta` (orange), `message_stop` (red). Small annotation next to `message_delta`: "contains stop_reason + usage".

When you open a streaming request, you always see events in this exact order.

`message_start` comes first — it carries message metadata like the model name and message ID.

[click]

Then `content_block_start` signals that a new block of content is beginning.

For a text response, this means a text block is opening.

[click]

`content_block_delta` is the workhorse — you'll receive dozens or hundreds of these.

Each one carries a small chunk of text in a field called `text_delta`.

[click]

`content_block_stop` closes the block when it's finished.

[click]

Then `message_delta` arrives — this is important for the exam.

It carries `stop_reason` and the final usage statistics.

[click]

Finally, `message_stop` signals the stream is completely done.

Memorize this sequence. The exam tests it.

## SLIDE 4: Streaming in Python — The SDK Way

**Visual**: Code editor panel on dark background with Python syntax highlighting. A second panel on the right shows terminal output with text appearing line by line, simulating live streaming output.

```python
import anthropic

client = anthropic.Anthropic()

# The SDK's stream() context manager handles SSE for you
with client.messages.stream(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Explain REST APIs in plain English."}]
) as stream:
    # text_stream yields each text chunk as a plain string
    for text in stream.text_stream:
        print(text, end="", flush=True)  # flush=True ensures immediate output

# After the context exits, the final message is available
final_message = stream.get_final_message()
print(f"\n\nStop reason: {final_message.stop_reason}")
print(f"Input tokens: {final_message.usage.input_tokens}")
```

The SDK's `stream()` context manager abstracts the SSE protocol entirely.

`text_stream` is a convenience iterator — it yields only the text deltas, nothing else.

You don't need to parse raw events just to display text.

[click]

After the `with` block exits, `get_final_message()` gives you the complete assembled message.

This is where you read `stop_reason` and token counts.

## SLIDE 5: Raw Event Streaming — When You Need Control

**Visual**: Code editor panel showing the raw event loop example. Each `event_type` is highlighted in a different color. A callout box points to `input_json_delta` with the note: "Used when streaming tool call arguments".

```python
import anthropic

client = anthropic.Anthropic()

# Use stream() when you need to handle every event type manually
with client.messages.stream(
    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "List three benefits of microservices."}]
) as stream:
    for event in stream:
        if event.type == "content_block_delta":
            # text_delta for text, input_json_delta for tool arguments
            if event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)
            elif event.delta.type == "input_json_delta":
                # Partial JSON from a tool call — accumulate this
                print(event.delta.partial_json, end="")
        elif event.type == "message_delta":
            # stop_reason is here, not in message_stop
            print(f"\nStop reason: {event.delta.stop_reason}")
```

When you're streaming tool calls, you need the raw event loop.

Tool arguments arrive as `input_json_delta` events — partial JSON that you accumulate.

Notice that `stop_reason` lives in `message_delta`, not `message_stop`.

This trips people up — including on the exam.

## SLIDE 6: Stream vs. No-Stream — The Decision Framework

**Visual**: Two-column comparison table with a clean header row. Left column: "Use Streaming" with a green checkmark icon. Right column: "Don't Stream" with a red X icon. Each column has four rows of bullet points. Background is light grey with subtle card styling.

Use streaming when your users are watching.

Chat interfaces, copilots, and assistants — anywhere a human is waiting — streaming makes these feel alive.

Long responses benefit most from streaming because the wait without it is the longest.

[click]

Don't stream when your code — not a human — is consuming the output.

Batch pipelines, classification jobs, and automated extraction tasks don't benefit.

You just need the final answer, not a token-by-token feed.

[click]

Don't stream when you need the *complete* response before you can do anything with it.

If you're parsing a full JSON structure, partial JSON is useless until it's done.

[click]

The rule of thumb: if a human sees the output in real time, stream it.

If a program consumes the output after it's complete, don't bother.

## SLIDE 7: Exam Tip

**Visual**: Amber callout box. Title: "Exam Tip". Icon: target/bullseye. Text: "Streaming changes delivery, not content. The exam tests perceived latency vs. total latency."

**Exam Trap**: Thinking that streaming makes Claude generate responses faster, or that it changes what Claude outputs.

**Correct Approach**: Streaming does not change the content or total generation time — it changes *when* you receive the first token. Streaming reduces perceived latency (time-to-first-token) but does not reduce total latency. The exam will present scenarios asking when streaming helps: the correct answer is always a user-facing interface where immediate feedback improves experience, not a batch pipeline. Also watch for questions about where `stop_reason` lives — it's in the `message_delta` event, not `message_stop`.

## SLIDE 8: Key Takeaways

**Visual**: Clean bullet list on light background. Title: "What to Remember".

- Streaming uses Server-Sent Events; the event order is always `message_start` → `content_block_start` → `content_block_delta` (many) → `content_block_stop` → `message_delta` → `message_stop`
- `stop_reason` and usage stats arrive in `message_delta`, not `message_stop`
- Streaming reduces time-to-first-token (perceived latency) but does not reduce total generation time
- Use streaming for user-facing interfaces; skip it for batch processing and programmatic pipelines that need the full response before acting
