## SLIDE 2: The Five Core Request Parameters

**Visual**: A Python code block on a dark background. Five parameters highlighted in different colors: `model` in blue, `max_tokens` in orange, `messages` in green, `system` in purple, `temperature` in red. Each has a short label floating beside it.

Every call to `client.messages.create()` shares the same skeleton.

There are five parameters you need to know cold.

`model` tells the API which Claude to use ‚Äî `claude-3-5-sonnet-20241022`, for example.

`max_tokens` sets the hard ceiling on how many tokens Claude can generate in the response.

`messages` is the conversation history ‚Äî an array of turns with roles and content.

`system` is where you put persistent instructions that apply to the whole conversation.

`temperature` controls how creative or deterministic the output is.

These five parameters are the foundation of everything else in this section.

## SLIDE 3: A Complete Request, Annotated

**Visual**: Dark-background code editor showing a full `client.messages.create()` call in Python. Each line has a comment on the right side. The `messages` array is expanded to show two items: one with `role: "user"` and one with `role: "assistant"`.

Let's look at a real, complete request.

```python
import anthropic

client = anthropic.Anthropic()  # Uses ANTHROPIC_API_KEY from environment

response = client.messages.create(
    model="claude-3-5-sonnet-20241022",   # Which Claude model to use
    max_tokens=1024,                       # Max tokens in the response
    system="You are a helpful assistant.", # Top-level system prompt
    messages=[
        {
            "role": "user",                # Must be "user" or "assistant"
            "content": "What is the capital of France?"
        },
        {
            "role": "assistant",           # Claude's previous turn
            "content": "The capital of France is Paris."
        },
        {
            "role": "user",               # The new question we're asking
            "content": "What is its population?"
        }
    ]
)
```

Notice a few things here.

`system` is at the top level of the call ‚Äî not inside the `messages` array.

The `messages` array alternates between `"user"` and `"assistant"` roles.

The conversation history is your responsibility to maintain and pass each time.

Claude has no memory between calls ‚Äî everything it knows about the conversation is in this array.

## SLIDE 4: The Response Object, Dissected

**Visual**: A formatted JSON object on a light background. Fields are shown in a two-column table: field name on the left, a one-sentence explanation on the right. Highlighted fields: `id`, `type`, `role`, `content`, `model`, `stop_reason`, `stop_sequence`, `usage`.

Now let's look at what comes back.

```python
# The response object looks like this:
{
    "id": "msg_01XFDUDYJgAACzvnptvVoYEL",  # Unique message ID
    "type": "message",                       # Always "message"
    "role": "assistant",                     # Always "assistant"
    "content": [                             # List of content blocks
        {
            "type": "text",                  # Content block type
            "text": "The population of Paris..."
        }
    ],
    "model": "claude-3-5-sonnet-20241022",  # Model that responded
    "stop_reason": "end_turn",              # Why Claude stopped
    "stop_sequence": None,                  # Which stop sequence triggered (if any)
    "usage": {
        "input_tokens": 42,                 # Tokens in your request
        "output_tokens": 31                 # Tokens in the response
    }
}
```

The `content` field is a list, not a string.

Even a simple text response is wrapped in a content block with `"type": "text"`.

That list structure matters ‚Äî when tool use is involved, you'll see `"type": "tool_use"` blocks instead.

`stop_reason` tells you *why* Claude stopped generating.

`end_turn` means Claude finished naturally.

`max_tokens` means it hit the ceiling you set.

`stop_sequence` means a custom stop string was triggered.

`usage` tells you exactly what you're being billed for.

## SLIDE 5: Reading the Response in Code

**Visual**: Python code snippet on dark background. Shows `response.content[0].text` being accessed. An arrow from the code points to the extracted string. A second snippet shows iterating over `response.content` with a for loop, with a comment "handles multiple content blocks."

Knowing the structure matters when you go to read the response.

```python
# Most common case: access the first text block directly
answer = response.content[0].text
print(answer)

# Safer: iterate over all content blocks
for block in response.content:
    if block.type == "text":
        print(block.text)

# Check why Claude stopped
print(response.stop_reason)   # "end_turn", "max_tokens", "stop_sequence", "tool_use"

# Check token usage
print(f"Input: {response.usage.input_tokens}")
print(f"Output: {response.usage.output_tokens}")
```

The shortcut `response.content[0].text` works for simple cases.

But in production ‚Äî especially with tools ‚Äî you should always iterate over `response.content`.

Checking `stop_reason` is important for robustness.

If `stop_reason` is `"max_tokens"`, the response was cut off and you may need to continue it.

## SLIDE 6: Exam Tip

**Visual**: Amber callout box. Title: "Exam Tip". Icon: target/bullseye. Text: "System is NOT a role ‚Äî it's a top-level parameter."

**Exam Trap**: Passing the system prompt as a message in the `messages` array using `role: "system"` ‚Äî this is not valid in the Claude API.

**Correct Approach**: The `system` parameter is always top-level in `client.messages.create()`, never inside the `messages` array. The only valid roles in `messages` are `"user"` and `"assistant"`.

If you see a question showing `{"role": "system", "content": "..."}` inside the messages array, that is the wrong answer. That pattern comes from OpenAI's API and does not work here.

---

## SLIDE 7: Key Takeaways

**Visual**: Clean bullet list on light background. Title: "What to Remember".

- Every `messages.create()` call requires `model`, `max_tokens`, and `messages` ‚Äî the `system` parameter is separate from the array
- The `messages` array only accepts two roles: `"user"` and `"assistant"` ‚Äî `"system"` is not a valid role
- Responses return a `content` list of typed blocks ‚Äî always `"text"` for plain responses, `"tool_use"` when tools are involved
- `stop_reason` tells you why generation ended ‚Äî always check it in production code to handle truncated responses
